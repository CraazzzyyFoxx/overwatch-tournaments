# Draft UX/UI Redesign — согласованный дизайн

**Дата:** 2026-07-14  
**Статус:** согласован  
**Область:** публичная live-доска, капитанский сценарий, зрительский режим, административная настройка и live control room

## 1. Контекст

Текущий драфт уже поддерживает создание и посев сессии, snake/linear/custom порядок, таймер, realtime-события, выбор игрока на роль, автопик, паузу, откат и экспорт команд. Однако публичный интерфейс, капитанский сценарий и административная настройка развивались как отдельные поверхности и не образуют единый безопасный процесс.

Ключевые проблемы текущей реализации:

- `DraftBoard.tsx` совмещает live-состояние, представление трёх ролей пользователей и админские действия в одном крупном компоненте.
- Страница использует параллельную систему `--draft-*`, hex-фон, несколько glow и condensed-uppercase типографику вместо актуальной схемы Editorial Tactical.
- Presence-события игнорируются, а online-статусы капитанов и число зрителей генерируются искусственно.
- Ошибки мутаций не объясняются пользователю; выбранный игрок очищается до подтверждения успешного ответа.
- Сервер проверяет только локальную вместимость роли текущей команды. Он не гарантирует, что после пика остальных игроков можно распределить по незаполненным ролям всех команд.
- Регистрационные notes попадают в публичный board snapshot.
- Админская настройка не показывает итоговый порядок, достижимость составов и последствия re-seed.
- Suggestions и override существуют на backend, но почти не представлены в UX.

## 2. Понимание задачи

- Главная цель live-интерфейса — помочь капитану быстро сделать допустимый и осознанный выбор под таймером.
- Зритель должен понимать ход драфта, не получая ложных данных или визуальных обещаний интерактивности.
- Администратор должен провести настройку, проверить готовность, запустить драфт и исправить исключительную ситуацию без прямого доступа к БД.
- Система не должна разрешать пик, после которого корректно завершить составы невозможно.
- Если нехватка роли всё же обнаружена, администратор может добавить роль игроку в snapshot текущей draft-сессии.
- Весь draft-домен переводится на актуальную дизайн-схему Editorial Tactical.
- Приватные организаторские данные и капитанская стратегия не публикуются зрителям.

## 3. Цели и не-цели

### Цели

- Пошаговая и проверяемая настройка драфта.
- Разные информационные и интерактивные режимы для капитана, зрителя и администратора.
- Глобальная гарантия достижимости role-составов.
- Аварийное исправление роли игрока из админки.
- Объяснимые suggestions и блокировки.
- Надёжный realtime UX с честными состояниями подключения.
- Полная визуальная миграция на `docs/design-book.md`.
- WCAG AA, полноценная клавиатура и отдельная мобильная информационная архитектура.

### Не-цели

- Замена текущего алгоритма балансировки.
- Автоматическое изменение исходной регистрации при аварийной правке draft snapshot.
- Публикация приватного shortlist капитана.
- Изменение общей административной навигации приложения.
- Добавление новых frontend UI-библиотек без необходимости.

## 4. Предположения и NFR

- До 12 команд и примерно 150 игроков в пуле.
- Board и pick-options должны обновляться визуально сразу после события.
- Целевой p95 для feasibility/options — до 300 мс на указанном масштабе.
- Realtime остаётся основным источником свежести, snapshot/refetch — механизмом сходимости.
- Драфт должен переживать reconnect, повторную доставку событий и конкурентные запросы.
- Организаторские notes приватны по умолчанию.
- Админские причины override/role edit должны храниться в admin-only аудите.
- Frontend проверяется lint-командой; `next build` для этой задачи не используется.

## 5. Выбранный продуктовый подход

Выбран подход **guided setup → live control room**.

До запуска администратор проходит мастер настройки. После запуска та же поверхность превращается в read-only summary и пульт управления. Публичная страница выбирает капитанский или зрительский режим по правам пользователя, не пытаясь показать один и тот же UI всем.

## 6. Административная настройка

### Шаг 1. Основные параметры

- Размер итогового состава.
- Число пиков команды по умолчанию выводится как `team_size - 1`, поскольку капитан уже занимает слот.
- Pick time с понятными пресетами и ручным значением.
- Snake, linear или custom с коротким визуальным примером.
- Autopick-стратегия с объяснением человеческим языком.
- Редкие параметры находятся в раскрываемом Advanced-блоке.

Если в будущем понадобится модель, где число раундов отличается от размера состава, это должно быть отдельным явно названным режимом с собственной серверной валидацией. Обычный draft не допускает рассинхронизацию этих полей.

### Шаг 2. Проверка пула

Показываются:

- общее число участников и требуемое число игроков;
- распределение primary/secondary/flex ролей;
- отсутствующие ранги;
- исключённые или неаппрувленные регистрации;
- игроки без связанного аккаунта;
- feasibility-результат и блокирующие дефициты.

### Шаг 3. Капитаны

- Поиск и фильтры по роли/рангу.
- Отдельный компактный список выбранных капитанов.
- Inline-редактирование имени команды.
- Закреплённый счётчик выбранных команд.
- Недопустимые количества объясняются до продолжения.

### Шаг 4. Порядок

- Для weakest/strongest/random показывается вычисленный порядок.
- Для manual используется drag-and-drop на уже установленном `dnd-kit`.
- Ниже показывается preview всех раундов и разворота snake.
- Random сохраняет seed, чтобы порядок был воспроизводим.

### Шаг 5. Review

Единый checklist проверяет:

- размер пула;
- число капитанов и команд;
- достижимость role-слотов;
- связность аккаунтов капитанов;
- итоговый порядок;
- таймер и autopick;
- приватность заметок;
- отсутствие блокирующих ошибок.

### Шаг 6. Ready/Start

- Ссылка на live-доску.
- Реальное состояние подключения капитанов.
- Финальное подтверждение запуска.
- Re-seed до старта показывает diff команд, игроков и пиков и требует подтверждения.

На desktop мастер использует форму слева и живой preview справа. На мобильном шаги идут одной колонкой с закреплёнными `Назад`/`Продолжить` и safe-area.

## 7. Live-доска капитана

Капитанский режим отвечает на три вопроса: чья очередь, кого можно выбрать и что изменит подтверждение.

### Ожидание

- Показывается число пиков до следующего хода команды.
- Капитан может собрать приватный shortlist.
- Уже выбранные другими командами игроки удаляются из shortlist, следующий кандидат поднимается выше.
- Shortlist хранится локально по `sessionId + authUserId`.

### Ход капитана

- Hero/статусная строка явно показывает команду, номер пика, незаполненные роли и таймер.
- Пул использует серверный список допустимых `(player, role)` options.
- Фильтр роли учитывает primary, secondary и flex.
- Заблокированный кандидат остаётся объяснимым: UI показывает, какой дефицит он создаст.
- Suggestions выводятся отдельной группой с объяснением роли, role-rank и причины fit.
- После выбора показывается review: игрок, выбранная роль, новый состав и оставшиеся role-слоты.
- Подтверждение доступно только при актуальной версии board/options.
- Выбор очищается только после успешного ответа.

При конфликте версии, уже выбранном игроке или потере соединения UI сохраняет подготовленный выбор, обновляет board и объясняет, что изменилось.

## 8. Зрительский режим

- Карточки/строки игроков не выглядят selectable.
- Главные элементы: последний пик, текущая команда, таймер, прогресс и изменения составов.
- Добавляется компактная публичная лента событий: pick, autopick, pause, resume, rollback, completion.
- Новое событие мягко подсвечивается через `transform/opacity`, без перестройки всей страницы.
- Suggestions, shortlist, причины админских действий и приватные notes отсутствуют.
- Если точного viewer count нет, метрика не показывается.

## 9. Live control room администратора

После старта мастер становится пультом управления:

- текущий пик, команда, таймер и realtime-состояние;
- реальный captain presence и время последней активности;
- feasibility-индикатор;
- public event feed и отдельный admin audit feed;
- Pause/Resume как основные действия;
- редкие опасные действия в отдельном меню.

Правила действий:

- Autopick сначала показывает кандидата, роль и причину.
- Rollback показывает конкретный отменяемый пик и предупреждает о переходе на паузу.
- Cancel требует усиленного подтверждения.
- Override требует preview, причину и показывает затронутую команду.
- Export показывает conflicts, progress, результат и безопасный retry.
- Двойные запросы блокируются до завершения текущего.

## 10. Глобальная достижимость ролей

### Модель

Feasibility строится как bipartite matching/max-flow:

- левая сторона — доступные игроки;
- правая сторона — незаполненные слоты `(draft_team_id, role)`;
- ребро существует, если игрок может легально играть роль;
- capacity игрока равна 1, capacity slot равна 1;
- flex связывается со всеми разрешёнными продуктом ролями;
- уже выбранные игроки и капитаны уменьшают соответствующие team-role capacities.

Драфт достижим, если максимальное паросочетание закрывает все оставшиеся слоты. Простого подсчёта игроков по ролям недостаточно: один flex не может одновременно закрыть два дефицита.

### Где выполняется проверка

1. Перед seed/ready и start — полная feasibility-проверка.
2. В `GET /picks/{pick_id}/options` — симуляция каждого легального `(player, role)` для текущего пика.
3. Внутри транзакции `select` — повторная authoritative-проверка выбранного варианта.
4. В autopick — выбор только среди safe options.
5. После role edit, rollback, override и re-seed — полный пересчёт.

Если выбор делает завершение невозможным, сервер возвращает `pick_makes_draft_infeasible` с безопасным структурированным объяснением. Капитан не может обойти эту блокировку.

### Если safe option отсутствует

Autopick не выбирает «любого доступного игрока». Сессия переводится в `paused`, публикуется `draft.blocked`, а control room показывает `role_shortage`. Отдельный статус `blocked` в lifecycle не вводится: это причина системной паузы, а не новый основной статус.

## 11. Аварийное добавление роли

В состоянии setup/ready/paused администратор может открыть `Resolve role conflict`:

- увидеть незаполняемые team-role slots;
- увидеть оставшихся игроков и их declared roles;
- выбрать игрока и добавить ему недостающую роль;
- указать role-rank либо явно подтвердить его отсутствие;
- обязательно указать причину;
- увидеть feasibility preview до сохранения.

Изменение применяется только к `DraftPlayerRole` snapshot текущей сессии. Исходная регистрация не меняется. После сохранения сервер:

- повторно проверяет feasibility;
- увеличивает версии затронутых данных;
- записывает admin-only audit event с before/after/reason;
- публикует публичное `draft.player_updated` без приватной причины;
- инвалидирует board/options/feasibility;
- позволяет администратору вручную возобновить драфт.

API:

```text
POST /api/balancer/draft/sessions/{session_id}/players/{player_id}/roles
```

Пример body:

```json
{
  "role": "support",
  "rank_value": 2875,
  "reason": "Регистрация не содержала фактическую вторичную роль игрока"
}
```

## 12. API и данные

### Новые reads

```text
GET /api/balancer/draft/sessions/{session_id}/feasibility
GET /api/balancer/draft/picks/{pick_id}/options
```

`feasibility` доступен администраторам во время setup и control room. `options` доступен только текущему капитану и администраторам.

`options` возвращает для каждого игрока допустимые роли, safe/blocked состояние, структурированную причину и необязательный suggestion score. Backend `select` не доверяет этому ответу и повторяет проверку транзакционно.

### Realtime

Добавляются/расширяются события:

- `draft.pick_made`: `target_role`, `target_rank_value`, `pick_version`;
- `draft.player_updated`;
- `draft.blocked`;
- реальные `draft.presence` данные.

Публичные события не содержат admin reason или приватные notes.

### Audit

Поскольку публичный `WorkspaceEvent` не подходит для приватной причины, вводится admin-only `DraftAuditEvent`:

- `session_id`;
- `actor_auth_user_id`;
- `action`;
- `entity_type/entity_id`;
- `reason`;
- `before_json/after_json`;
- `created_at`.

Server state хранится в React Query. Realtime обновляет cache либо инвалидирует узкий query. Фильтры пула живут в URL; выбранный игрок и открытые панели — локальное transient-state; shortlist — приватное локальное состояние.

## 14. Editorial Tactical migration

Миграция охватывает happy path, loading, empty, error, reconnecting, paused, blocked, completed и mobile.

- Удаляется самостоятельная семантика `--draft-*`; используются глобальные `--aqt-*` токены.
- Верх страницы строится через `PageHero/HeroFrame`, `HeroCoord`, `HeroStat`, `HeroStamp`.
- Hex-фон и несколько glow заменяются masked square grid, одним teal hairline и одним спокойным teal glow.
- Заголовки — mixed-case Onest; UI — проектный Inter; телеметрия — JetBrains Mono.
- Значимый текст не мельче 11 px.
- `Air over boxes`: открытые группы и hairline-разделители вместо вложенных карточек.
- Role colors используются только семантически; интерфейсный акцент один — teal.
- Общая ширина — `max-width: 1400px`.
- CSS Module остаётся только для геометрии timer ring и ограниченных realtime-анимаций.

### Responsive

- Три колонки только на широких экранах.
- На ноутбуке DraftOrder становится collapsible sidebar.
- На мобильном используются вкладки `Пул`, `Моя команда`, `Порядок`.
- Таймер остаётся в компактной статусной строке.
- Pick review и confirm открываются как bottom sheet с safe-area.
- Не используется `100vh`; высота строится через `dvh`/естественный flow.

### Accessibility

- `aria-pressed` для выбора игрока и фильтров.
- Спокойный `aria-live` для смены хода, success и errors, но не для каждой секунды таймера.
- `focus-visible` 2px teal на всём интерактиве.
- Профиль игрока — настоящая ссылка, не click-handler внутри кнопки.
- Состояния не кодируются только цветом.
- Touch targets не меньше 44×44 px.
- `prefers-reduced-motion` отключает декоративные переходы.

## 15. Ошибки и edge cases

- Выбранный игрок сохраняется при сетевой/API-ошибке.
- `player_unavailable` и `pick_already_resolved` вызывают refetch и понятное объяснение.
- Потеря realtime блокирует confirm до получения актуальной версии.
- Reconnect восстанавливает snapshot и replay cursor.
- Rollback освобождает игрока, пересчитывает feasibility и оставляет draft paused.
- Role edit невозможен во время live без паузы.
- Re-seed невозможен после start.
- Export показывает idempotent result/retry, не создавая команды повторно.
- Filter-zero state содержит причину и `Сбросить фильтры`.
- Viewer count и presence не подменяются фиктивными значениями.

## 16. Тестирование

### Backend

- Недостаточный пул.
- Достаточное число role labels, но невозможное matching-распределение.
- Один flex претендует на несколько критичных слотов.
- Safe и unsafe hypothetical picks.
- Role edit восстанавливает feasibility.
- Concurrent select с expected version.
- Autopick выбирает только safe option.
- Нет safe option → paused + `draft.blocked`.
- Permissions и admin-only audit.
- Приватные notes отсутствуют в board/events.

### Frontend

- Setup wizard validation и review.
- Selection сохраняется после ошибки.
- Options блокируют unsafe choice и показывают причину.
- Reconnect/stale state.
- Captain, spectator и admin режимы.
- Desktop/notebook/mobile.
- Полный keyboard flow и screen-reader announcements.
- WCAG AA, reduced motion, touch targets.
- 150 игроков/12 команд без заметной деградации.

## 17. Rollout

1. Feasibility работает в shadow mode и логирует потенциальные нарушения.
2. Админский preflight показывает предупреждения без hard-block.
3. После проверки реальных данных включается серверный hard-block для новых сессий.
4. Добавляется аварийный role edit и admin audit.
5. Включается новая Editorial Tactical страница и control room.
6. Старый интерфейс остаётся под feature flag на один турнирный цикл.
7. После успешного цикла удаляются старые `--draft-*` и монолитные компоненты.

## 18. Decision Log

| Решение | Альтернативы | Причина |
|---|---|---|
| Guided setup → control room | Одна улучшенная форма; полностью свободная режиссёрская панель | Лучший баланс безопасности, понятности и стоимости |
| `rounds = team_size - 1` для обычного драфта | Независимые поля | Капитан уже входит в состав; независимые значения создают неразрешимые состояния |
| Max-flow feasibility перед каждым пиком | Только счётчики ролей; только аварийный override | Учитывает flex/secondary и гарантирует завершимость |
| Hard-block unsafe pick для капитана | Только warning | Warning не защищает последний пик |
| Safe-only autopick | Fallback на любого игрока | Fallback может нарушить role capacity и целостность состава |
| System pause при role shortage | Новый lifecycle status `blocked` | Меньше состояний; причина паузы передаёт нужную семантику |
| Role edit только в draft snapshot | Автоматически менять регистрацию | Аварийное действие не должно скрыто менять источник данных |
| Admin-only audit table | Причина в публичном WorkspaceEvent | Не раскрывает приватную причину зрителям |
| Разные captain/spectator режимы | Единая доска с условно скрытыми кнопками | Разные задачи требуют разной информационной иерархии |
| Полная Editorial Tactical migration | Локальный рескин текущего CSS | Draft не должен поддерживать параллельную дизайн-систему |
| Mobile tabs + bottom sheet | Вертикально сложенные desktop-панели | Уменьшает scroll cost и сохраняет критичный таймер/confirm |
| Shadow rollout перед hard-block | Немедленное включение | Позволяет проверить алгоритм на реальных турнирах |

## 19. Критерии готовности

- Администратор может создать, проверить, запустить и завершить draft без прямой правки БД.
- Ни один капитанский пик не может сделать корректное завершение составов невозможным.
- При ошибочной роли администратор добавляет её игроку из control room с preview и аудитом.
- Публичный API не содержит приватных notes или admin reasons.
- Captain, spectator и admin UI используют честные realtime-состояния.
- Вся draft-поверхность соответствует Editorial Tactical и WCAG AA.
- Критические backend/frontend сценарии покрыты тестами.
