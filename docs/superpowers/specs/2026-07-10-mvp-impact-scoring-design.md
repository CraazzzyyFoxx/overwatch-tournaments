# MVP Impact Scoring — дизайн

Дата: 2026-07-10
Статус: утверждён (вариант C: гибрид, старые поинты сохраняются)
Исследование: https://claude.ai/code/artifact/fe4eb64b-8ad1-4ef2-bfd0-82bd17f31478

## 1. Контекст и мотивация

Текущий MVP (`parser-service/src/services/match_logs/flows.py`) — одна линейная
свёртка на всех игроков:

```
PerformancePoints = Elims*500 + FinalBlows*250 + Assists*50
                  + HeroDamage + Healing - Deaths*750 + DamageBlocked*0.1
```

Анализ по 4 236 матчам с килл-фидом (турниры 36–73) показал:

- **Ролевой перекос ×2.65**: танки — 20% игроков, но 52% наград MVP (одна
  шкала на роли с разными «вселенными чисел»: медианный танк — 8 936 урона за
  10 мин, саппорт — 2 728).
- **Ранговый перекос**: верхний квинтиль по рангу забирает 52% MVP, нижний — 5.9%.
- **Мёртвые слагаемые**: FinalBlows 5.9% вклада, Assists 2.1%, DamageBlocked 0.5%.
- **Вин-сигнал живёт в элиминациях и смертях**: логит-регрессия победы по ролям
  (acc 0.82–0.86) даёт элимам +1.8…+2.7, смертям −1.5…−2.2; сырой объём урона
  и хила при фиксированных элимах — отрицательный предиктор.
- Плоские бонусы за события поверх сырой шкалы (эксперимент V1) не работают:
  91.7% тех же MVP.

Эксперименты V2 (per-role z-score) и V4 (V2 + вин-веса) полностью чинят ролевой
перекос (×1.01–1.14) при сохранении связи с победой (69.9% MVP из победившей
команды); V3 (базлайны роль × ранг) дополнительно чинит ранговый перекос
(Q5: 52% → 28%) ценой смены семантики награды.

## 2. Цели

1. Официальный MVP матча — новый **ImpactPoints (V4)**: честный к ролям,
   учитывающий события (first picks, киллы с ульты, мультикиллы и т.д.).
2. Бейдж **«Сверх ожиданий» (V3)** — отдельная награда за игру сильно выше
   ожиданий от своего ранга.
3. **PerformancePoints/Performance сохраняются без изменений** — считаются и
   отображаются как раньше («классические очки»). Сайт показывает и то и то.
4. Бэкфилл всей истории (~6.9k матчей).

### Не-цели

- Изменение/удаление текущей формулы PerformancePoints.
- ML-модель вместо интерпретируемой формулы.
- Пер-герой impact-скоринг (только per-round и per-match, hero_id NULL).
- Realtime-пересчёт базлайнов.

## 3. Семантика наград

| Метрика | Смысл | Где рисуется |
|---|---|---|
| `Performance` / `PerformancePoints` | «классические очки», как сейчас | таблицы статистики матча, колонка «Очки (классика)» |
| `ImpactPoints` / `ImpactRank` | официальный MVP: лучший игрок с поправкой на роль | gold/silver/bronze пилюли (`mvpRank()` переезжает на `impact_rank`) |
| `OverperformanceScore` | насколько игрок превзошёл ожидания своего ранга | бейдж «Сверх ожиданий»: топ-1 матча по скору И скор ≥ порога (константа, старт 2.0) |

## 4. Новые статы из kill_feed

При обработке лога (и в бэкфилле) дополнительно считаются и пишутся в
`matches.statistics` новые члены `LogStatsName`:

| Стат | Определение |
|---|---|
| `FirstPicks` | киллер первого килла каждого файта (группировка kill_feed по (round, fight), минимум по time) |
| `FirstDeaths` | жертва первого килла файта |
| `UltimateKills` | киллы с `ability = 'Ultimate'` |
| `SupportKills` | киллы, где роль героя жертвы — support |

Self-kill'ы (killer_id == victim_id) исключаются. Как и остальные derived-статы:
строки per-round (round > 0) и агрегат round = 0, `hero_id NULL`. Отображаются
на сайте как обычные статы (leaderboards, страница матча).

Примечание: `Enum(LogStatsName)` персистит ИМЯ члена — raw SQL/индексы пишутся
по именам (`'FirstPicks'`), не по значениям.

## 5. Скоринг

Общий конвейер для ImpactPoints и OverperformanceScore:

1. **Роль игрока в матче** — по доминирующему `hero.type` по HeroTimePlayed;
   fallback — declared role из `tournament.player.role`.
2. **Темп**: каждый стат → значение за 10 минут игрового времени
   (`value / HeroTimePlayed * 600`); игроки с < 60 сек игрового времени
   скорятся как 0.
3. **z-оценка**: `(rate - mean) / std` по лиговому базлайну; ImpactPoints —
   базлайн (role, stat); OverperformanceScore — базлайн (role, rank_bucket, stat).
   Обрезка (winsorize) до ±3.
4. **Композит**: `score = Σ w_s * z_s * time_share`, где
   `time_share = минуты игрока / max минут в матче`.
5. **Ранжирование**: `ImpactRank` = 1..N по убыванию ImpactPoints внутри
   раунда и матча (та же механика, что у Performance).

Стартовые веса (из экспериментов, версия формулы `impact_v1`):

```
eliminations 1.3   finalblows 0.4    deaths -1.3
herodamagedealt 0.35   healingdealt 0.35   damageblocked 0.25
offensiveassists 0.45  defensiveassists 0.45  ultimatesused 0.1
multikills 0.45  solokills 0.35  objectivekills 0.3  environmentalkills 0.2
first_picks 0.55  first_deaths -0.45  ult_kills 0.5  support_kills 0.3
```

Веса живут в конфиг-модуле parser-service вместе с `FORMULA_VERSION`.

**Гейт килл-фида**: для матчей без kill_feed событийные z принудительно = 0
(«нет данных» ≠ «сыграл плохо»). Признак — отсутствие строк kill_feed по матчу.

## 6. Базлайны

Новая таблица `matches.stat_baselines`:

| колонка | тип | смысл |
|---|---|---|
| `formula_version` | str | версия формулы/весов (напр. `impact_v1`) |
| `role` | Enum(HeroClass) | роль |
| `rank_bucket` | int NULL | NULL — базлайн V4; 0..2 — терциль ранга для V3 |
| `stat` | Enum(LogStatsName) | стат |
| `mean`, `std` | float | параметры распределения rate-per-10min |
| `meta` | JSONB | границы ранг-бакетов (замораживаются в версии), n наблюдений |
| `computed_at` | timestamptz | |

- Пересчёт — джоба в parser-service (typed RPC + CLI), запуск вручную или после
  турнира. Источник — те же per-match агрегаты (round=0, hero NULL), матчи с
  килл-фидом для событийных статов, HeroTimePlayed ≥ 3 мин.
- Границы ранг-бакетов — терцили `tournament.player.rank` на момент пересчёта,
  сохраняются в `meta` (скоринг читает границы из строки, не пересчитывает).
- Parser читает актуальную версию с кэшем (cashews; `configure_cache()` в
  entrypoint воркера — известный gotcha).
- Первичный сид — тем же пересчётом сразу после миграции.

## 7. Пайплайн и бэкфилл

**Онлайн-путь**: в `create_stats()` после существующих derived-статов —
шаг «события из kill_feed» (kill_feed-строки уже создаются этим же пайплайном)
и шаг «impact-скоринг» (нужны базлайны из кэша).

**Бэкфилл** — idempotent-джоба parser-service (RPC + CLI):

1. Для каждого матча: читает готовые строки `matches.statistics` + kill_feed.
2. Удаляет свои прежние строки (новые статы + скоры) по match_id.
3. Считает и вставляет батчами.

Прогон по ~6.9k матчей; повторный запуск безопасен. Нужен также при смене
весов/базлайнов (новая formula_version → бэкфилл по желанию).

## 8. API (app-service)

В match/user-эндпоинтах рядом с `performance` появляются:

- `impact_rank: int | null` — место по ImpactPoints (пилюли);
- `impact_points: float | null`;
- `overperformance: { score: float, badge: bool } | null` — badge = топ-1 матча
  и score ≥ порога.

Read-path — те же выборки из `matches.statistics` по новым именам статов.

## 9. Frontend

- `mvpRank()` и пилюли (MvpMatchPill и др.) переключаются на `impact_rank`.
- `performance` остаётся в типах и таблицах — колонка «Очки (классика)».
- Бейдж «Сверх ожиданий» — отдельная пилюля с тултипом
  («ранг N — сыграл как топ-X своей роли»).
- Новые статы (FirstPicks, UltimateKills, …) появляются в таблицах статистики
  автоматически по мере добавления в отображаемые наборы.
- Детали UI — на этапе плана имплементации.

## 10. Edge cases

- **Матчи без килл-фида** (турниры < 36): событийные статы не пишутся,
  событийные z = 0, ImpactPoints считается по остальным статам.
- **< 60 сек игрового времени**: скор 0, ранг в хвосте.
- **12 игроков в матче** (подмены): ранжирование по всем участникам, time_share
  дисконтирует короткие выходы.
- **Ничьи по скору**: детерминированный tie-break (меньший user_id — как в
  текущем `cumcount`-ранжировании).
- **Нет базлайна** (новый стат/роль): z = 0, warning в лог.
- **std = 0**: z = 0.

## 11. Тесты

- Unit: детектор first pick/first death (группировка по файтам, минимум по
  времени, self-kill исключён); z-композит на фиксированных базлайнах
  (включая winsorize, time_share, std=0); гейт «нет килл-фида → события не
  штрафуют»; badge-порог.
- Integration: полный прогон тестового лога → новые строки статистики +
  корректный ImpactRank; idempotence бэкфилла (двойной прогон = те же строки).

## 12. Риски и ограничения

- Базлайны от версии к версии меняют скоры задним числом только при явном
  бэкфилле — иначе история остаётся на старой версии (это осознанно).
- MercyRez-событий в данных нет — резы Мерси не учитываются.
- Kill_feed есть только с 36-го турнира — для старых турниров impact считается
  без событийной части (сравнимость внутри турнира сохраняется).
- Расхождение «старый MVP ≠ новый MVP» в ~40% исторических матчей — ожидаемо,
  коммуницируется как фича (обе метрики видны).
